# AGENTS.md

このファイルは、ジユウノコンパスを改善するCodex/AIエージェント向けの恒久方針です。
今後このリポジトリで作業する前に、必ずこのファイルを読み、ここに書かれた思想・表現・実装方針に従ってください。

## ジユウノコンパスの目的

ジユウノコンパスは、日本在住者が「生活を壊さずに、働き方の自由度を少しずつ上げる」ための生活コンパス型診断サイトです。

主役は、FIRE達成判定ではありません。
生活費、収入、貯金、投資額、固定費、仕事のしんどさ、働き方の調整余地をもとに、今の現在地と次の一歩を短くわかりやすく示すことが目的です。

ユーザーが結果を見たあとに、次のように感じられる状態を目指します。

- 何をするサイトかすぐわかる
- ざっくり入力でも使えそうだと思える
- 生活防衛資金と資産カバー率の現在地がわかる
- 働き方を緩める現実的な選択肢が見える
- 次にやることが1〜3個に絞られている
- 退職、投資、副業を煽られている感じがしない

## 守るべき思想

- やさしく、落ち着いて、現実的に背中を押す。
- 「人生を一気に変える」ではなく「まず一歩整える」。
- 不安を煽らず、生活の余白を増やす道案内にする。
- 退職、投資、転職、副業を断定的にすすめない。
- FIREは最終目的地のひとつとして扱い、入口や主役にしない。
- 数字は判断材料であり、人生の正解や保証として見せない。
- 「辞めるか続けるか」の二択ではなく、在職で負荷を下げる、固定費を整える、準備を進める、少し働くなどの中間案を出す。
- 既存の文章・ブロック・順序・UI・診断ロジックは仮のものとして扱い、プロダクト改善として必要なら自律的に変える。

## FIRE/取り崩し表現の注意

4%ルールを「安全な答え」として扱わないでください。
4%ルールは米国の過去データに基づく参考線であり、日本在住者の主判定には使いすぎないでください。

表示する場合は、以下のように扱います。

- 2.5%: 保守的な目安
- 3.0%: 標準寄りの目安
- 3.5%: やや攻めた目安
- 4.0%: 米国過去データの参考線

避ける表現:

- あなたはFIRE達成です
- 4%なら安全です
- NISAなら税金ゼロです
- iDeCoも入れればもう自由です
- S&P500なら長期で勝てます

推奨する表現:

- 資産が生活費の何%を支えられるかの目安です
- 生活費の一部を資産で補える状態です
- 4.0%は米国過去データの参考線です
- iDeCoは老後資産として有力ですが、原則60歳まで引き出せないため、早期の生活費原資とは分けて考えます
- 海外株インデックスは為替や外国課税の影響を受けます

FIREやセミリタイアは、「完全に働かない」ではなく「生活費の一部を資産や安定収入で支え、働き方を調整しやすくする」考え方として見せます。

## 生活防衛資金の扱い

生活防衛資金は単一の正解金額として出さず、レンジと段階で表示します。

基本レンジ:

- 会社員・公務員: 生活費の3〜6か月分
- フリーランス・自営業・収入変動が大きい人: 生活費の6〜12か月分
- 子どもあり、単一収入、高家賃、大きなローン、教育費が重い場合: 上限寄り

貯金が少ないユーザーには、いきなり6か月分を求めません。
次のような段階目標を使います。

- 1万円
- 3万円
- 10万円
- 生活費0.5か月分
- 生活費1か月分
- 下限レンジ
- 安心寄りレンジ

表示は「不足しています」だけにせず、次のような前向きな進捗表現にします。

- いまは、はじめの足場づくりです
- 次は3万円を目指すと、小さな想定外に強くなります
- 下限レンジまであと○円です
- 生活防衛資金は投資資金とは分けて考えます

生活防衛資金の計算では、原則としてすぐ使える現預金を主に見ます。投資資産、iDeCo、将来の年金を防衛資金として強く当て込まないでください。

## 固定費改善の優先順位

固定費改善は、自由度を上げる重要ミッションとして扱います。
ただし「節約しろ」と迫るのではなく、「毎月の固定費を1つ軽くすると、働き方を選びやすくなる」と見せます。

優先順位:

- 住居費
- 保険料
- 通信費
- サブスク
- 光熱費
- ローン関連

住居費や保険は効果が大きい一方で着手が重いため、次のように分けて表示します。

- 今日できる小さな改善: 使っていないサブスクを1つ止める、通信オプションを確認する
- 今週できる改善: 通信プランを比較する、固定費を一覧化する
- 今月見る大きな改善: 保険、家賃、住宅ローン、車、教育費、返済計画を点検する

固定費シミュレーターやミッションでは、「大きく効く改善」と「今日できる改善」を両方出してください。

## 働き方を緩める提案方針

「辞める」「FIREする」を最初の提案にしないでください。
働き方を緩める提案は、原則として低リスクな順に出します。

推奨順序:

1. 有給を取る
2. 残業を整理する
3. 在宅勤務・時差出勤を相談する
4. 業務量や締切を調整する
5. 異動・職種変更を検討する
6. 時短勤務・週4勤務・短時間正社員を検討する
7. 転職準備を在職中に進める
8. 契約社員・派遣・パートなど、雇用形態を軽くする
9. 休職制度や相談窓口を確認する
10. 退職は最後の選択肢として扱う

仕事のしんどさが強い人には、副業やフリーランス化を最初に出さないでください。
まず休息、制度確認、相談、業務負荷の軽減を優先します。

副業やフリーランス化は、疲弊が落ち着いていて、生活防衛資金や時間の余白がある人にだけ慎重に提示します。

## 診断ロジック方針

ジユウノコンパスは単なる資産診断ではなく、生活と働き方のコンパスです。
診断では、お金と働き方の両方を組み合わせます。

考慮する軸:

- 月生活費
- 現預金
- 投資額
- 毎月の積立・黒字額
- 生活防衛資金の月数
- 仕事のしんどさ
- 残業・通勤・対人負荷
- 在宅勤務や時短など、社内で緩められる余地
- 副収入や世帯収入の有無
- 転職準備度
- 家族構成や固定費の重さ

結果タイプ例:

- まず生活防衛タイプ
- 固定費見直しタイプ
- 休息・相談優先タイプ
- 在職で負荷軽減タイプ
- 転職準備先行タイプ
- 資産補完で働き方調整タイプ
- セミリタイア準備タイプ

結果画面は、原則として次の順に整理します。

1. あなたの現在地
2. いま強いポイント
3. 注意したいポイント
4. 資産が生活費を支える割合
5. 生活防衛資金の達成度
6. 働き方を緩める余地
7. 今日・今週・今月のミッション
8. 参考情報・免責

## UI/UX方針

- 初見ユーザーが、このサイトで何ができるかすぐわかるようにする。
- 入力ハードルを低くする。最初はざっくりでよい。
- 詳細入力は任意にし、「わからない場合は空欄でもOK」と伝える。
- 結果画面は文章を長く並べず、カード、メーター、短文、ミッションで構成する。
- 次にやることは1〜3個に絞る。
- スマホで読みやすく、カード内の文字が詰まりすぎないようにする。
- ユーザーを焦らせる赤・警告・ランキング表現を使いすぎない。
- 派手なSaaS風、ゲーム風、副業煽り風にしない。
- 生活コンパス、地図、現在地、道しるべ、防壁、余白などの静かなモチーフを使う。
- カードの入れ子を避け、ページ全体を落ち着いた情報設計にする。
- ボタンや入力は、何を入力するか、手取りか税込みか、二重入力にならないかがわかる文言にする。

## アイコン・メーター・図解の使い方

アイコンは装飾ではなく、理解を助けるために使います。
使える視覚表現:

- 生活防衛資金の達成度メーター
- セミリタイア距離メーター
- 自由度スコアのゲージ
- 資産が生活費の何%を支えるかのカード
- 結果タイプごとのアイコン
- ミッションカードのアイコン
- 「現在地 → 次の一歩」のステップ図
- 「お金の土台」「働き方」「次の一歩」の3分割カード

アイコンは既存の `lucide-react` を優先して使います。
新しいライブラリは、必要性が高い場合だけ追加してください。

## 禁止事項

- FIREを煽るサイトにしない。
- 投資、退職、転職、副業を断定的にすすめない。
- 金融助言、法律助言、医療助言のように見える表現にしない。
- 4%ルールを絶対安全なものとして扱わない。
- 日本の税制、年金、NISA、iDeCo、社会保険について断定しすぎない。
- 新NISAを万能と書かない。
- iDeCoを早期リタイア用の生活費原資として強く扱わない。
- 年金額を自動で強く当て込まない。
- Deep Researchの長文をそのまま貼り付けない。
- 架空の制度名や根拠不明の数値を追加しない。
- 不安を煽る赤字・不足・危険表示に寄せすぎない。
- 派手なSaaS風、ギラギラした副業サイト風、過度なゲーム風にしない。
- 技術構成やディレクトリ構成を必要以上に大きく変えない。
- 既存機能を不用意に壊さない。
- 不要なライブラリを増やしすぎない。

## 実装時の進め方

1. まず `AGENTS.md`、`README.md`、必要に応じて `PRODUCT_PRINCIPLES.md` と `IMPROVEMENT_DECISIONS.md` を読む。
2. FIRE、取り崩し、生活防衛資金、固定費改善、働き方を緩める提案に関わる変更では、該当する `docs/research/` の研究資料を読む。
3. 現在のコード構成、診断ロジック、主要コンポーネントを確認する。
4. 変更が、生活コンパスの目的とこの方針に合うか確認する。
5. 文章だけで済ませず、必要なら診断ロジック、UI構成、入力項目、ミッション、メーターを改善する。
6. ただし、1PRで確認できる範囲にまとめる。
7. 新しい機能は最小実装から入れる。
8. 既存コードの構成に合わせ、過剰な抽象化は避ける。
9. 新しいライブラリは原則追加しない。既存のReact、TypeScript、Tailwind、lucide-react、Rechartsを優先する。
10. 変更後、利用可能なチェックを実行する。
   - `npm test`
   - `npm run lint`
   - `npm run build`
11. フロントエンドの見た目や操作を変えた場合は、ローカルで画面確認する。

## 作業開始時のルール

複雑な改善タスクでは、実装前に必ず短い作業計画を作成してください。

計画には以下を含めます。

- 今回の1PRで扱う範囲
- 今回は扱わない範囲
- 優先順位
- 影響する主要ファイル
- 実行予定のチェック

計画後は、1PRで確認可能な範囲に絞って実装してください。
計画は実装を止めるためではなく、改善範囲を決めて自律的に進めるためのものです。

## コミット方針

自律改善でコミットを作る場合は、変更を意味のある単位で分けます。

- 1PR内では2〜5コミット程度を目安にする。
- 小さい修正だけなら1コミットでよい。
- 10コミットを超えそうなら、PR分割を検討する。
- コミットは「あとから戻したい単位」で分ける。
- 壊れた状態、仮実装、デバッグログあり、unrelatedな変更が混ざった状態ではコミットしない。
- コミット前に、可能な範囲で `npm run lint`、`npm test`、`npm run build` を確認する。
- コミットメッセージはConventional Commitsに近い形式にする。
  - 例: `feat: add emergency fund range meter`
  - 例: `fix: soften withdrawal copy`
  - 例: `docs: update agent guidance`
- 作業完了時は、作成したコミット一覧と各コミットの目的を報告する。

ユーザーの既存作業や未コミット差分がある場合は、混ぜてコミットしないでください。
同じファイルに自分の変更と他者の変更が混在する場合は、差分を確認し、必要ならコミット前に範囲を説明します。

## Commit autonomy

ユーザーは差分の詳細を判断できない前提で、Codexが `git status` と `git diff` を確認し、意味のある単位でコミット可否を判断してください。

コミットしてよい場合:

- 差分が同じ目的の変更として説明できる。
- 変更内容が `AGENTS.md` の方針に沿っている。
- 壊れた状態や仮実装ではない。
- 関連するチェックが通っている、またはチェック不能な理由を説明できる。
- コミットメッセージと変更理由を明確に説明できる。

コミットせず止める場合:

- unrelatedな差分が混ざっていて安全に切り分けられない。
- 変更の目的が説明できない。
- ビルド、lint、testが失敗している。
- 仮実装やデバッグコードが残っている。
- ユーザー確認が必要な設計判断が含まれている。

判断に迷う場合はコミットしないでください。
その代わり、以下を報告してください。

- 未コミット差分があるファイル
- それぞれの差分の推定目的
- コミットできない理由
- ユーザーに確認したいこと
- 推奨する次の操作

## 作業完了時の報告形式

作業完了時は、簡潔に以下を報告します。

- 変更内容
- 追加した機能
- Deep Researchや本方針をどこに反映したか
- 判断理由
- 実行したチェック
- 未対応項目
- 次に改善するとよいこと

ドキュメントだけの変更でテストやビルドを省略した場合は、その理由を明記してください。
